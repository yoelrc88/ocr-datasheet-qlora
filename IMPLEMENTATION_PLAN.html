olmOCR Datasheet Table Fine-Tuning Plan

A first-pass execution plan for fine-tuning allenai/olmOCR-2-7B-1025 on hard datasheet tables, scoped for a small adapter run on a 12 GB GPU such as an RTX A2000.

Base: allenai/olmOCR-2-7B-1025 Method: QLoRA 4-bit NF4 Stage 1: SFT Focus: merged-cell tables

Goal

Improve datasheet table extraction for hard structure failures such as merged headers, rowspan, colspan, multi-line cells, footnotes, rotated tables, and vendor-specific layouts.

Scope

Keep the first run narrow and adapter-based. This is a targeted table-structure adaptation project, not a broad OCR upgrade.

First Deliverable

A smoke-tested adapter that beats baseline on a small unseen eval set of datasheet tables.

Implemented Now

The repo already includes a Colab notebook, a training entrypoint, an eval helper, and dataset examples so the first smoke test can start immediately.

How Drive Works

The notebook does not pull data from GitHub automatically. It expects your repo and your dataset to exist inside mounted Google Drive, then it copies the repo into Colab's local disk for faster execution.

Drive is optional now. For smoke tests, you can skip Drive entirely: the notebook can clone the repo from GitHub into Colab and generate mock data locally.

Expected Drive Layout

  • /content/drive/MyDrive/finetune-test as the repo root inside Drive
  • /content/drive/MyDrive/finetune-test/scripts/ for the training scripts
  • /content/drive/MyDrive/finetune-test/examples/ for example dataset files
  • /content/drive/MyDrive/finetune-test/reports/ for lightweight run outputs that you will commit back to GitHub

What The Notebook Reads

  • PROJECT_DIR: the repo folder in Drive
  • DATA_JSONL: the labeled dataset manifest
  • each JSONL row must point to a real image path available from the Colab runtime

In GitHub-only mode, PROJECT_DIR is the cloned repo under /content/ocr-datasheet-qlora instead of a Drive path.

What The Notebook Writes

  • temporary training artifacts into Colab local disk under /content/artifacts/<run_name>
  • lightweight reports into Drive under reports/runs/<run_name>

That split is intentional: local disk is faster for training, Drive is used for durable small files that are safe to push back to GitHub.

If you are not using Drive and are only saving the executed notebook back to GitHub, remember that files created inside the Colab runtime are not automatically committed. The notebook now prints the run-report contents so notebook-only saves are still reviewable.

What Data Colab Expects

The training script expects a JSONL dataset where each row contains an image path and a gold output. For the current starter path, the gold output is HTML.

Minimum Row Shape

  • id
  • doc_id
  • image: absolute path to an image file visible in Colab
  • html: the gold table HTML
  • notes: optional comments

Important Detail

The script validates that every image path exists before training starts. If the repo path or image paths are wrong, the run will fail before the GPU does useful work.

Cheap Smoke Test

You do not need a full real dataset just to prove the notebook works. The fastest smoke test is to create a tiny mocked dataset and verify that the pipeline can read files, stage the repo, start validation, and either begin training or fail for a clearly model-specific reason.

Best Smoke-Test Goal

  • prove Drive paths are correct
  • prove the JSONL is readable
  • prove the image paths resolve
  • prove the training script starts from Colab

Recommended Mock Setup

  • use 2-4 tiny PNG images
  • use very small fake table labels in valid HTML
  • set MAX_SAMPLES to a tiny number
  • run only a smoke-test configuration first
  • use scripts/generate_mock_smoke_test_data.py to generate a starter mocked dataset in the repo

Cheapest Colab Path

  • set USE_DRIVE = False
  • set DATA_MODE = 'mock' for a plumbing-only check
  • or set DATA_MODE = 'synthetic' for the next real experiment path
  • let the notebook download the repo from GitHub
  • let the notebook generate the chosen dataset locally in Colab

If the repo is private, GitHub clone mode needs a token or a switch back to USE_DRIVE = True.

What Counts As Success

  • the notebook finds PROJECT_DIR
  • the notebook copies the repo into /content/finetune-test
  • the script writes run_config.json and dataset_split_summary.json
  • the run produces a report folder under reports/runs/<run_name>

For a pure infrastructure smoke test, you do not care yet whether the model learns anything useful.

Synthetic Data

The mock smoke-test data is only for plumbing. Synthetic data is different: it can be useful for actual training if you generate the exact structure failures you want the model to learn, such as merged headers and merged body cells.

What We Can Generate Well

  • header cells with rowspan and colspan
  • body cells merged across rows or columns
  • small changes in cell width, row height, and padding
  • slight shifts in text position and layout proportions
  • repeated table families with controlled variation

Why It Is Interesting

For your use case, synthetic generation is a strong fit because the target skill is structural table recognition, not open-ended world knowledge. If the model keeps failing on merged-cell layouts, we can manufacture many clean examples of those layouts cheaply.

Current Starter Generator

  • scripts/generate_synthetic_merged_table_data.py
  • outputs paired PNG images and gold HTML labels
  • randomizes widths, heights, text content, and merged-cell patterns
  • intended to run in Colab, not to commit generated images into git
  • default next-step notebook mode: DATA_MODE = 'synthetic' with about 150 generated samples

Important Limitation

Synthetic tables can teach structure, but they will not fully cover real datasheet rendering artifacts by themselves. The strongest setup is usually synthetic data for structure coverage plus a smaller set of real cropped tables for realism.

Constraints

  • Use allenai/olmOCR-2-7B-1025 BF16 as the fine-tuning base.
  • Do not fine-tune a Q8 inference artifact.
  • Keep the existing olmOCR rendering and prompting pipeline intact for baseline compatibility.
  • Start with SFT, not RL or GRPO.
  • Default to QLoRA 4-bit NF4.

Success Criteria

The first pass is useful if at least one of these improves on held-out documents.

  • Exact table structure match
  • rowspan and colspan correctness
  • Row and column count consistency
  • Numeric fidelity for units, signs, and decimals
  • Output validity for HTML or JSON

Recommended Task Shape

Start Here

  • Focus on merged-cell datasheet tables first.
  • Prefer HTML if structure fidelity is the main problem.
  • Use JSON only if downstream tooling already expects normalized structured tables.

Avoid First

  • General OCR improvement goals
  • Full-page training before crop-level proof
  • Markdown as the first target for merged-cell cases

Dataset Plan

First Pass

  • 100-300 hard examples
  • Prefer cropped table regions over full pages
  • Group examples by source document
  • Hold out unseen documents for validation and test

Expansion Pass

  • 300-500 high-value examples if the smoke test moves the target metric

Required Sample Schema

  • id
  • doc_id
  • image
  • html or json
  • notes

Labeling Rules

  • Keep labels strict and consistent
  • Do not mix target formats in one training run
  • Reject ambiguous gold structure
  • Preserve merged-cell structure exactly in the label

Baseline Before Training

Run stock olmOCR-2-7B-1025 through the normal olmOCR toolkit path before training anything.

Measure errors by bucket, not by general impression.

  • Merged headers
  • Row and column span errors
  • Multi-line cell flattening
  • Row split or row merge errors
  • Decimal, sign, and unit mistakes
  • Reading order mistakes
  • Invalid HTML or JSON

Training Plan

First-Pass Configuration

  • Base model: allenai/olmOCR-2-7B-1025
  • Method: QLoRA
  • Quantization: 4-bit NF4
  • Batch size: 1
  • Gradient accumulation: enabled
  • Gradient checkpointing: enabled
  • Image size: roughly 768-1024px longest side for training
  • Output format: one strict format only

LoRA Scope

  • Start narrow with q_proj and v_proj
  • Freeze most of vision on the first pass
  • Expand only if the adapter is clearly too weak

Runtime Strategy

  • Run a 1 epoch smoke test first
  • Confirm VRAM stability before longer runs
  • Prefer short validation intervals
  • Save predictions for manual review, not just loss curves

Evaluation Plan

Core Metrics

  • Exact structure match when possible
  • Predicted vs gold row count
  • Predicted vs gold cell count
  • rowspan count match
  • colspan count match
  • Valid HTML or JSON
  • Numeric fidelity checks on held-out samples

Manual Review Buckets

  • Invented rows or cells
  • Dropped rows or headers
  • Span collapse
  • Footnote leakage into cells
  • Header and footer contamination
  • Ordering mistakes

Known Risks

  • Fine-tuning alone will not fix poor rendering or bad table localization.
  • Full-page training is heavier and may hide whether crop-level learning works.
  • Long target outputs slow training and increase memory pressure.
  • A small dataset can help a narrow vendor or table family but will not create a general OCR upgrade.
  • Random page-level splits can leak document structure across train and eval.
